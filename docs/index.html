Core Design

Per-Partition Raft Consensus

The architectural decision that separates TurboMQ from every other JVM-based message broker.

Unlike Kafka's KRaft mode where a single controller cluster manages leader election for all partitions across the entire broker cluster, TurboMQ runs an independent Raft group per partition — drawing direct inspiration from CockroachDB's per-range Raft implementation.

Each partition is a fully autonomous consensus unit. Leader election in partition A is completely orthogonal to partition B — no shared controller, no coordination bottleneck, no cluster-wide blast radius.

This architecture unlocks capabilities that a centralized controller can never provide: partitions can be atomically migrated between brokers using a learner-catch-up-promote sequence without any consumer-visible interruption.

  • Partition leader election is fully independent — no cluster-wide impact
  • Partitions migrate between brokers without stopping reads or writes
  • Failure isolation at the partition level, not the broker level
  • Horizontal scale without re-sharding the controller bottleneck

Kafka / KRaft

Single Controller Cluster

  • One controller manages all partition leaders
  • Leader election requires controller round-trip
  • Controller failure = cluster-wide impact
  • Partition migration requires full stop
  • Scale ceiling on controller throughput
TurboMQ

Per-Partition Raft Groups

  • Each partition elects its own leader
  • Election is local, no cluster coordination
  • Broker failure affects only its partitions
  • Live migration via learner promotion
  • Linear scale — add brokers, gain throughput

Component Deep Dive

Every subsystem engineered from first principles. Click any card to expand the full implementation detail.

⚡
Raft Consensus
raft module
▼

A complete Raft implementation with pre-vote protocol to prevent disruptive elections from network-partitioned nodes. Log replication uses batched AppendEntries RPCs, and the commit index advances only when a quorum acknowledges the entry.

InstallSnapshot enables lagging replicas to catch up via RocksDB checkpoint transfer rather than replaying the full log — critical for zero-downtime partition migration where a learner node must reach parity before promotion.

Pre-vote protocol Log replication Commit index tracking InstallSnapshot RPC Learner mode Election timeout jitter
📡
SWIM-lite Gossip
network module
▼

A SWIM-inspired gossip protocol for membership and failure detection. Nodes probe peers with direct pings; on timeout, k indirect probes are issued through randomly selected intermediaries to distinguish network partition from node failure.

Incarnation numbers prevent stale "dead" broadcasts from overwriting live state — a node that receives a dead-declaration about itself immediately increments its incarnation and refutes the claim through gossip dissemination.

Direct ping Indirect probe (k=3) Suspect state Dead state Incarnation-based merge Self-refutation
🗄
RocksDB Storage
storage module
▼

Each partition is backed by a dedicated RocksDB column family, providing physical data isolation without the overhead of separate database instances. WriteBatch groups all append operations into atomic commits, ensuring that log entry durability is all-or-nothing.

Checkpoint snapshots use hard links — RocksDB's native checkpoint API creates a point-in-time consistent view with near-zero overhead, enabling the InstallSnapshot path to transfer a full partition state to a learner node.

Column family per partition WriteBatch atomicity Checkpoint snapshots Hard-link efficiency SST compaction
⚖
Partition Manager
broker module
▼

Responsible for initial placement and ongoing rebalancing of partitions across brokers. Round-robin placement distributes new partitions evenly. A greedy rebalancer detects skew and schedules migrations to converge the cluster.

Zero-downtime migration follows a four-phase protocol: add the target as a Raft learner → wait for catch-up (log gap below threshold) → promote to voter → remove the source replica. Consumers experience zero interruption because the leader changes only after full replication.

Round-robin placement Greedy rebalancer Learner promotion Catch-up detection Zero-downtime migration
👥
Consumer Groups
broker module
▼

The ConsumerGroupCoordinator manages group membership via heartbeat sessions. When a consumer joins or leaves, a rebalance is triggered: the coordinator collects the current member list and applies the configured assignment strategy to re-distribute partitions.

Both Range and RoundRobin strategies are supported. Range assigns contiguous partition ranges per consumer for locality; RoundRobin maximizes evenness. Session expiry is tracked per-member — a missed heartbeat window triggers a coordinated group rebalance without impacting other groups.

Join/leave rebalance Heartbeat sessions Range assignment RoundRobin assignment Session expiry
✓
Exactly-Once Delivery
broker module
▼

Idempotent producers are assigned a stable producerId at session establishment. Each message carries a monotonic sequence number scoped to the (producerId, partition) pair. The broker deduplicates retries by checking the sequence against the last-committed sequence for that producer.

This guarantees exactly-once semantics at the storage layer without distributed transactions — a retry of a message that was already committed is silently acknowledged rather than double-written.

Idempotent producer producerId assignment Sequence deduplication Per-partition tracking Retry-safe

Real-World Applications

When to Use TurboMQ

01

Event-Driven Microservices

Decouple services with durable, ordered event streams. Order service publishes "OrderPlaced" → inventory, payment, and shipping services consume independently. Per-partition ordering guarantees events for the same order are processed sequentially.

02

Real-Time Data Pipelines

Ingest clickstreams, IoT sensor data, or application logs at 50K+ msg/sec per node. Consumer groups enable parallel processing across workers. RocksDB persistence means no data loss even during consumer downtime.

03

Financial Event Sourcing

When every state change must be recorded as an immutable event — banking transactions, audit trails, CQRS systems. Exactly-once delivery via idempotent producers ensures no duplicate or lost transactions, even during broker failover.

04

Zero-Downtime Scaling

Add or remove brokers without stopping producers or consumers. Per-partition Raft means partitions migrate independently — no cluster-wide rebalancing storm. SWIM gossip automatically discovers new brokers within seconds.

05

Multi-Tenant SaaS Isolation

Assign each tenant their own partitions with independent Raft groups. A noisy tenant's partition leader election doesn't affect other tenants. Virtual-thread-per-partition architecture handles 10K+ partitions without thread pool exhaustion.

06

Change Data Capture (CDC)

Stream database changes to downstream systems — search indexes, caches, data warehouses. Consumer group rebalancing ensures processing continues even when workers crash. Snapshot transfer catches up slow followers without blocking the leader.
